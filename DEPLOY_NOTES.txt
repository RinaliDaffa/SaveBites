SaveBites — Supabase Deployment Notes
======================================

Project URL: https://dbizcmezzdsusqymagln.supabase.co
Dashboard:   https://app.supabase.com/project/dbizcmezzdsusqymagln/editor


1. Files in this drop
---------------------
  supabase/schema.sql                       Base schema (profiles, listings,
                                            orders, reviews, RLS, triggers).
  supabase/migration-add-geo-columns.sql    Adds last_lat / last_lng on profiles
                                            for consumer geo-discovery.
  supabase/migration-add-business-fields.sql
                                            Adds business_name, address, lat,
                                            lng on profiles for merchants.
  scripts/deploy-schema.sh                  Bash deployer (CLI preferred, falls
                                            back to manual SQL Editor steps).
  .env.example                              Template for .env.local.


2. Required environment variables
---------------------------------
  NEXT_PUBLIC_SUPABASE_URL      Supabase project URL (public).
  NEXT_PUBLIC_SUPABASE_ANON_KEY Anon key (public; RLS-restricted).
  SUPABASE_SERVICE_ROLE_KEY     Service role key (server-only, bypasses RLS).

  All three must be present in .env.local. The service role key is required
  for Server Actions to insert listings on behalf of merchants without
  hitting RLS restrictions — createServerClient in actions.ts uses it for
  mutations.


3. Deploy order
---------------
  a) Apply supabase/schema.sql first (only needed on a fresh database).
     It creates profiles, listings, orders, reviews, RLS policies, triggers,
     and the nearby_listings RPC.

  b) Apply supabase/migration-add-geo-columns.sql.
     Adds last_lat, last_lng + index. Idempotent (IF NOT EXISTS).

  c) Apply supabase/migration-add-business-fields.sql.
     Adds business_name, address, lat, lng + supporting indexes. Idempotent.

  d) If the Supabase CLI is installed and the project is linked, run:
       bash scripts/deploy-schema.sh
     Otherwise paste each .sql file into the Supabase Dashboard → SQL Editor.


4. Verify after deploy
----------------------
  Run these in the SQL Editor:

    -- 4a. Column presence on profiles
    select column_name, data_type
    from information_schema.columns
    where table_schema = 'public'
      and table_name   = 'profiles'
    order by ordinal_position;

    -- Expected: id, role, full_name, last_lat, last_lng, business_name,
    --           address, lat, lng, created_at, updated_at.

    -- 4b. nearby_listings RPC exists
    select proname from pg_proc where proname = 'nearby_listings';
    -- Expected: 1 row.

  If 4b returns 0 rows, the RPC is missing. Run the definition from
  supabase/schema.sql (search for "create or replace function public.nearby_listings").
  The function uses ll_to_earth() from the cube + earthdistance extensions,
  which schema.sql enables.


5. RLS notes
------------
  - profiles: any authenticated user can SELECT; only the row owner can UPDATE.
  - listings: any authenticated user can SELECT active rows; merchants manage
              their own rows.
  - orders:   consumer and merchant on the order can SELECT/UPDATE.
  - reviews:  any authenticated user can SELECT; only the review's consumer
              can INSERT/UPDATE.

  Service role key bypasses RLS — keep it server-only. Do NOT prefix it with
  NEXT_PUBLIC_.


6. Quick smoke test (post-deploy)
---------------------------------
  1. Sign up a merchant via the app; confirm a row appears in profiles with
     role = 'merchant' and business_name, address, lat, lng populated.
  2. Create a listing via the merchant dashboard; confirm the row appears
     in listings with status = 'active'.
  3. As a consumer, hit the discover page; confirm nearby_listings returns
     the new listing when within the radius.


7. Rollback
----------
  Migrations are additive (ADD COLUMN IF NOT EXISTS, CREATE INDEX IF NOT
  EXISTS). To roll back, drop the added columns manually:
    alter table public.profiles
      drop column if exists last_lat,
      drop column if exists last_lng,
      drop column if exists business_name,
      drop column if exists address,
      drop column if exists lat,
      drop column if exists lng;
  Drop the indexes too if you created them:
    drop index if exists public.profiles_last_location_idx;
    drop index if exists public.profiles_business_name_idx;
    drop index if exists public.profiles_merchant_geo_idx;
